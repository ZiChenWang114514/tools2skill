# Portability Rules — What Makes a Skill Exportable

A skill generated by tools2skill is exportable only if it can be installed on any machine
and used by any Claude Code user without modification. This document defines the rules.

---

## Rule 1: No Absolute Paths

**Forbidden:**
```
Run: /home/zcwang/myenv/bin/python script.py
Config at: /data2/zcwang/tools2skill/config.yaml
```

**Required:**
```
Run: python script.py  (from project root)
Config at: ./config.yaml  (relative path)
```

**Why:** Absolute paths are machine-specific. On another machine, `/home/zcwang/` does not exist.

**t2s check:** Pattern `/home/\w+` and `/data\d*/\w+` are flagged by `t2s validate`.

---

## Rule 2: No Hardcoded Usernames

**Forbidden:**
```
Connect as user 'zcwang'
Upload to: s3://zcwang-bucket/
```

**Required:**
```
Connect as: $USER (or specify your username)
Upload to: s3://<your-bucket>/
```

**Why:** Usernames are personal identifiers. Exported skills are public artifacts.

---

## Rule 3: No Secrets or Credentials

**Absolutely forbidden:**
```
API key: sk-abc123...
Password: hunter2
Token: ghp_...
```

**Required:**
```
API key: $OPENAI_API_KEY (set via environment variable)
```

**Why:** Skills may be published to public registries or shared openly.
Any secret in a skill file is a security incident.

**t2s check:** Patterns like `api_key\s*=\s*\w{8,}` are flagged.

---

## Rule 4: No Machine-Specific Environment Assumptions

**Forbidden:**
```
Uses conda env at /data/zcwang/anaconda3/envs/membrane-vhppi
Requires CUDA 12.4 (specific to this machine's GPU)
```

**Required:**
```
Requires: conda env with torch>=2.0  (or: pip install torch)
GPU: CUDA 11.8+ recommended (CPU fallback available)
```

**Why:** Other users may have different environments. Skills should describe
*requirements*, not *local facts*.

---

## Rule 5: Version Ranges, Not Pinned Versions

**Prefer:**
```
Requires: Python ≥ 3.10, pytest ≥ 7.0
```

**Avoid:**
```
Requires: Python 3.11.4, pytest 7.4.3 (exact pins)
```

**Exception:** If a specific version is genuinely required due to a breaking change,
note it explicitly with the reason:
```
Requires: requests == 2.28.x  [WARNING: 2.29+ changed SSL behavior — see GOTCHA #2]
```

---

## Rule 6: No Local File References in Code Examples

**Forbidden:**
```python
df = pd.read_csv("/data2/zcwang/results/output.csv")
```

**Required:**
```python
df = pd.read_csv("your_data.csv")  # replace with your file path
```

---

## Rule 7: Parameterize Everything Machine-Specific

When a pattern genuinely varies by machine/user, use explicit placeholders:

```bash
# TEMPLATE — replace <your-host>, <your-port>, <your-user>
ssh <your-user>@<your-host> -p <your-port> "command"
```

---

## Portability Checklist (for Phase 4 — before export)

Before running `t2s export`, verify:

- [ ] No absolute paths in skill.md
- [ ] No usernames (mine or anyone's) in skill.md
- [ ] No API keys, tokens, or passwords
- [ ] All conda/venv references use env names, not absolute paths
- [ ] Code examples use placeholder filenames, not real data paths
- [ ] Version requirements stated as ranges unless breakage-specific
- [ ] `t2s validate <name>` passes with 0 violations

---

## Portability Status in Manifest

The manifest tracks portability status:

```json
"portability": {
  "machine_specific": false,
  "requires_local_setup": false,
  "notes": ""
}
```

Set `machine_specific: true` if the skill genuinely cannot be made portable
(e.g., wraps a proprietary internal tool). Such skills can still be exported
but will carry a warning label in `t2s list`.
